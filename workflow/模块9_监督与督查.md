# 模块9 · 监督与督查（监督层 / 项目传感器）

> 使命：在长周期开发中持续探测"实际进展 vs 既定目标"的偏移，及时纠偏、防遗忘、判断能否进入下一关卡。
> 定位：横跨所有模块的监督层，不生产功能，只做"体检"与"评审"。
> 理念来源：Harness Engineering 的"传感器+反馈"——发现偏移就触发自我纠正。
> 本产品范例：座位排布应用

## 一、核心信条
- 长项目最大的敌人不是难题，是"不知不觉地跑偏和遗忘"。
- 进度不靠记忆，靠看板；推进不靠感觉，靠关卡标准。
- 宁可定期停下来花10分钟体检，也不要闷头做错三天。

## 二、它回答的五个关键问题（与你的需求一一对应）
| 你的担忧                     | 本模块对应机制                 |
| ---------------------------- | ------------------------------ |
| 1 开发目标偏移               | 目标对齐检查（每次体检第1项）  |
| 2 忘记整体进度               | 进度看板（始终最新，一眼看全） |
| 3 当前阶段是否达标、能否推进 | 关卡评审会（Gate Review）      |
| 4 发现前序任务需返工         | 返工触发与回插机制             |
| 5 任务超目标完成、可提前推进 | 超额完成判定（允许跳步）       |

## 三-A、真实状态判定规程（Agent 每次启动必做）

> 目标：防止把模板范例误判为真实进度。判断进度必须靠证据，不靠感觉、不靠示例文字。

### 1. 必查证据
Agent 开始推进任何任务前，必须先检查以下证据，并向用户报告结果：
- **项目结构证据**：根目录是否已有产品代码目录、依赖文件、启动脚本、测试脚本。
- **需求证据**：是否有已填写的真实需求验证报告，且包含目标用户、访谈/抱怨证据、验证结论。
- **调研证据**：是否有已填写的竞品对比、用户抱怨高频词、一句话差异化定位。
- **规格证据**：是否有当前功能的正式 SPEC，且不是模板范例，包含功能目标、范围边界、行为规格、验收标准。
- **任务证据**：是否有经用户确认的 TASKS，拆成界面/数据/逻辑微任务，并有依赖和验收标准。
- **实现证据**：是否存在对应代码、测试、运行记录、真实环境验证记录或 DoD 勾选证据。
- **沉淀证据**：如果本轮遇到坑/弯路，是否已记录到实践知识库并写明反写动作。

### 2. 判定规则
按下面顺序从早到晚判断；一旦某项证据缺失，就停在对应关卡/模块：

| 证据状态 | 当前应处位置 | 下一步互动方式 |
| --- | --- | --- |
| 没有工具链、Demo、一次完整流程证据 | 关卡0 · 打基础 | 先确认本机工具链和最小 Demo 跑通计划 |
| 没有真实需求验证报告 | 关卡1 · 需求与方案锁定 / 模块1 需求识别与验证 | 引导用户填写假设、目标用户、访谈对象和验证结论 |
| 有需求但没有竞品调研与差异化定位 | 关卡1 · 需求与方案锁定 / 模块2 市场调研 | 引导用户做竞品表、抱怨词和一句话定位 |
| 有需求和调研，但没有正式 SPEC | 关卡1 到关卡2 的入口 / 模块3a SPEC | 先写当前功能 SPEC，不写代码 |
| 有 SPEC，但没有经确认的 TASKS | 关卡2 · MVP可用 / 模块3b TASKS | 只输出微任务清单，等用户确认 |
| 有 TASKS，但没有测试或实现证据 | 关卡2 · MVP可用 / 模块3 IMPLEMENT | 按单个微任务 TDD：先测试，再最小实现，再验收 |
| 有实现但缺测试、真实环境验证或 DoD | 关卡2 · MVP可用 / 模块3c DoD | 补测试、补真实环境验证、补验收记录 |
| MVP 闭环真实跑通且 DoD 通过 | 关卡3 · 上架就绪 / 模块4 营销与成交 | 准备上架素材和冷启动测试 |
| 已有种子用户但反馈机制不足 | 关卡4 · 营销成交 / 模块5 客户管理 | 建反馈入口、反馈表和数据看板 |

### 3. 输出格式
Agent 每次完成状态判定后，必须用下面格式向用户说明：

```text
状态判定：
- 已检查证据：____
- 已存在证据：____
- 缺失证据：____
- 当前关卡：____
- 当前模块：____
- 下一步建议：____
```

### 4. 保守原则
- 若证据不足，选择更早的关卡/模块。
- 模板、示例、占位符、空白表格，不算完成证据。
- 用户口头说明可作为线索，但关键进度仍应尽快沉淀到对应文档。
- 判定结果和用户认知冲突时，先复述冲突点，请用户确认，再推进。

## 四、定期体检（解决"目标偏移"，建议每完成1-2个功能做一次）

> 用下面这份清单自查，或贴给AI让它扮演"项目督查"帮你过一遍。

### 体检清单
- [ ] **目标对齐**：现在做的功能，是否仍指向SPEC的"功能目标"和模块2的"差异化定位"？有没有偷偷加了没用的功能？
- [ ] **范围守边**：是否越过了SPEC第3节"明确不做什么"的边界？
- [ ] **进度真实**：看板上勾的"完成"，是否都真在真实环境跑通过（而非AI口头完成）？
- [ ] **质量底线**：已完成功能是否都过了DoD？有没有"先欠着"的测试或坑？
- [ ] **节奏健康**：是否卡在某个任务超过预期很久？卡住了是否该求助或拆更小？

### 给AI的督查指令
> 你现在是我的项目督查。这是我的进度看板和当前SPEC。请对照检查：
> 1) 我当前在做的事是否偏离了原定目标和范围边界？
> 2) 有没有我标了"完成"但实际证据不足的任务？
> 3) 基于现状，你建议我下一步做什么、先别做什么？
> 直接指出问题，不要客气。

## 五、关卡评审会（解决"能否推进下一阶段"）

> 每当你觉得一个关卡快完成时，开一次"评审会"（自己对照，或让AI主持）。

### 评审规则
- 逐条核对该关卡《路线图》里的"通关标准"，全部为真才放行。
- 任一条存疑 → 不通关，列出补救任务，回到执行层。
- 通关后：更新看板状态，正式进入下一关卡，并在知识库记一句"本关最大收获/教训"。

### 评审记录模板
- 评审关卡：关卡2 MVP
- 逐条结论：标准1 ✅ / 标准2 ✅ / 标准3 ❌(测试未覆盖换座) / 标准4 ⬜
- 结论：[ ] 通关 [x] 不通关，补救任务：为换座逻辑补测试
- 下一步：完成补救后重审

## 六、返工触发与回插机制（解决"前序任务需返工"）

> 开发中途发现前面做错/要改时，用这套流程，避免"越改越乱"。

1. **不立刻动手改**，先在知识库《踩坑/弯路》记一条：为什么要返工？
2. **评估影响面**：让AI分析"改这个会牵连哪些已完成任务"。
3. **把返工当成一个新微任务**插入TASKS清单（标注"返工"+依赖），排进执行顺序。
4. **返工完成后重跑相关测试**，确认没有把别处改坏（回归验证）。
5. **反写**：若是规格没写清导致的返工 → 回改SPEC模板，防止再犯。

### 返工登记
| 序号 | 返工对象     | 原因           | 影响的任务 | 状态   |
| ---- | ------------ | -------------- | ---------- | ------ |
| 1    | 座位数据结构 | 没预留长桌字段 | T3,T5      | 进行中 |

## 七、超额完成判定（解决"超目标完成可提前推进"）

> 不是所有超额都是好事，用三问判断该不该跳步/进下一阶段。

- **问1 真达标了吗**：超出的部分是否仍满足DoD（有测试、真环境跑通）？还是只是"看起来做完了"？
- **问2 有没有透支**：是否为了快，欠下了测试/文档/坑没记？欠了就先补。
- **问3 下一步就绪吗**：提前进入的下一阶段，它的输入条件是否已具备？
- 三问皆"是" → 更新看板，允许跳步并记录"本任务超额完成，提前进入X"。
- 任一为"否" → 先补齐债务再推进。

## 八、与其它模块的衔接
- 数据来源：读取各模块的TASKS、DoD、路线图关卡标准。
- 输出去向：偏移和返工记录 → 回流《99 实践知识库》；评审结论 → 更新路线图关卡状态。
- 建议频率：日常体检随手做；关卡评审在每关收尾时做；返工随时触发。
