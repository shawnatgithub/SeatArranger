# 模块1_需求识别与验证

> 使命：在写一行代码前，确认这个需求真实存在、真的痛、真有人愿意用甚至付费。
> 通关标准（对应关卡1）：痛点经过至少5个真实对象验证，产出一页纸验证报告。
> 本产品范例：轻量AI辅助的座位排布应用（会议 / 宴请两种场景）

## 一、核心信条
- 需求未验证通过，不准进入市场调研和开发。
- 验证用最低成本完成：先聊天、看抱怨，再考虑做东西。
- 宁可证伪一个假设，也不要带着幻想开发三个月。

## 二、假设清单（先把想法写成可被证伪的句子）

> 格式：我相信【某类人】因为【某原因】会愿意【付费/使用】我的【某功能】。

- **核心假设**：我相信【经常要安排会议/宴席座位的人，如行政、宴会策划、婚礼司仪】，因为【手动用Excel或纸笔排座又慢又容易错、改一次要重排】，会愿意【使用】我的【点选+拖拽快速排座工具】。
- **付费假设**：我相信【其中的专业人士（如婚庆、宴会策划）】会愿意【付费】，因为【它直接节省他们的工作时间】。
- **风险假设（最该先验证的）**：我担心【普通人一年只排几次座，频率太低，不愿专门用工具】。

## 三、目标用户画像
| 类型     | 谁                      | 痛在哪                 | 使用频率 | 付费意愿 |
| -------- | ----------------------- | ---------------------- | -------- | -------- |
| 高频专业 | 婚庆/宴会策划、活动执行 | 反复改座、客户临时变动 | 高       | 较高     |
| 中频职场 | 公司行政、会务          | 会议席位+领导排位讲究  | 中       | 中       |
| 低频个人 | 办婚宴/家宴的个人       | 一次性、怕坐错得罪人   | 低       | 低       |

## 四、最低成本验证方法（按成本从低到高）
- [ ] **看抱怨**：在小红书/知乎/贴吧搜"排座位 麻烦 / 婚宴 座位表 工具"，记录真实抱怨原话
- [ ] **聊真人**：找至少5个目标对象（最好含1-2个专业策划）聊：你现在怎么排？最烦哪一步？愿意用新工具吗？
- [ ] **看替代方案**：他们现在用什么凑合（Excel？纸？某个App？）——能凑合说明痛，凑合得很好说明不痛
- [ ] **假门测试（可选）**：做一张产品介绍图发出去，看有没有人问"哪里能用"

## 五、访谈记录区（原话最有价值，别只记结论）
- 对象1（身份____）：现在用____排座，最烦____，对新工具反应____
- 对象2：……
- （至少5条）
