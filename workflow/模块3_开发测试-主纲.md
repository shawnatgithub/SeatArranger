# 模块3 _开发测试_（核心瓶颈模块）-主纲

> 融合三大理念：规格驱动(SDD) + 先设计微任务TDD(Superpowers) + 犯错即固化(Harness/Loop)
> 本产品范例：轻量AI辅助的座位排布应用（会议 / 宴请两种场景，点选+拖拽）

## 一、开发主流程（规格四步法）

```
1. SPECIFY 规格  → 写清做什么、验收标准（见 3a_SPEC.md）
2. PLAN    方案  → 定技术栈、架构、数据结构
3. TASKS   拆解  → 拆成可验收微任务（见 3b_TASKS.md）
4. IMPLEMENT 实现 → 逐个微任务：先测试→再代码→验收（见 3c_DoD.md）
```
铁律：四步顺序不可跳。没有规格不写计划，没有计划不拆任务，没有任务不写代码。

## 二、四个引擎

### 引擎1 · 目标制定（分层 + 验收标准）
- 里程碑目标（关卡级）：可演示。例：「能拖拽布置一桌宴请座位并保存」
- 任务目标（半天级）：可测试。例：「实现圆桌组件，可放置最多12个座位点」
- 每个目标都必须附"验收标准"，否则不准开工。

### 引擎2 · 任务分解（漏斗式四步）
对每个新功能依次做：
1. 写成用户故事：作为__，我想要__，以便__
2. 列三类清单：界面 / 数据 / 逻辑
3. 拆成单一职责微任务（AI一轮能做完、你10分钟能验收）
4. 标注依赖与验收标准，排执行顺序
→ 这一步先让AI"只输出清单不写代码"，你审核后再执行。

### 引擎3 · 成果验证（三层）
- 功能验证：对照验收标准亲手点一遍
- 自动化测试：AI写测试，每次改动后跑一遍防止改坏
- 真实环境验证：在浏览器/真机里真跑，不信AI口头"已完成"

### 引擎4 · 自动沉淀（犯错即固化，Loop核心）
- 每解决一个bug/走一个弯路 → 记进 99_实践知识库.md
- 每周把对话丢给AI生成复盘草稿 → 审核归档
- 关键：把教训反向写回 SPEC模板 / DoD清单，让错误不再重演

## 三、给AI的开发会话启动语（每次开新会话先贴）

> 我们用规格驱动方式开发。当前任务见 SPEC 文件。请严格遵守：
> 1. 先复述你对需求的理解并提出澄清问题，不要立刻写代码。
> 2. 按"界面/数据/逻辑"把功能拆成微任务清单，等我确认。
> 3. 每个微任务：先写失败的测试 → 写最小实现让测试通过 → 对照验收标准说明。
> 4. 若遇到已知坑，先查《实践知识库》是否已有解法。
> 5. 完成后提醒我是否需要记录新教训。
