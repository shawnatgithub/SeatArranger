# 快速排布座位（会议版）MVP PRD（微信小程序优先）

版本：MVP v1.0  
目标用户：年轻职场人（行政/业务/项目负责人等）  
核心价值：在不熟悉座次规则且时间紧迫时，**快速生成可执行的会议座次示意图**，并支持手动调整与导出。

---

## 1. 结论先行（已确认的关键决策）

### 1.1 MVP范围
- 先做：**会议排座**
- 形态：**自动推荐 + 手动调整（拖拽/交换/锁定）**
- 平台：**先微信小程序**（架构预留后续适配小红书小程序）
- 人员复用：**项目内一次性**（不做常用通讯录/组织架构）
- 导出：**PNG图片（优先）**
- 导入：**V1支持CSV导入，并提供导入模板**

### 1.2 账号与付费（MVP即接入）
- 账号：需要登录
- 付费形态：
  - **订阅：月 + 年**
  - **按次：按导出计费**（每次导出高清无水印付费）
- 免费版限制（你提出并确认可行方向）：
  - **“快速排座”增加排队时长：10–20秒**
  - **不支持导入人员（仅手动录入）**
  - （建议的补充，可选其一叠加以提高可解释性）导出图片加水印/仅标清、模板数量限制

> 说明：仅用“排队时长”做限制会带来口碑风险，建议至少叠加一个“可解释限制”（如水印/标清/模板数），让用户理解“为何要付费”。

## 2. 用户与使用场景

### 2.1 典型场景
1) 客户到访汇报会：需安排主宾与我方负责人座次，并考虑投影屏与入口引导  
2) 上级视察会议：上级主位明确，陪同、汇报人位置有约束  
3) 政府接待座谈：礼仪敏感，宁可“保守推荐 + 可解释提示 + 手动确认”

### 2.2 核心成功指标（MVP）
- TTV（Time to Value）：从进入小程序到导出座次图 ≤ 2 分钟
- 自动推荐可用率：用户只需≤3次手动微调即可可执行（可通过事件埋点衡量）
- 导出成功率：≥95%

## 3. 端到端用户流程（MVP）
1. 登录 → 进入项目列表  
2. 新建项目：选择会议类型/接待情景（策略）  
3. 选择场地模板 → 进入画布  
4. 确认/拖拽定位元素（门/主席位/投影屏…）  
5. 人员录入（手输或CSV导入）  
6. 点击「快速排座」→ 生成推荐座次  
7. 手动调整（拖拽/交换/锁定/撤销重做）  
8. 导出PNG（免费：标清/水印/排队；付费：高清无水印）

## 4. 功能模块与需求（验收标准导向）

### 4.1 项目模块（Project）
**功能点**
- 项目列表：创建、打开、重命名、删除/归档
- 项目配置：会议策略、模板类型、画布尺寸、是否开启吸附

**验收标准**
- 项目可在登录态下云端保存/多端恢复（至少同一微信账号）
- 删除有二次确认；归档可恢复

---

### 4.2 场地排布（Canvas + Template）
**MVP模板（至少6个）**
- 长桌（双侧座）
- 双排（面向投影）
- U型
- 回字形
- 主席台（台上/台下）
- 圆桌讨论（会议圆桌，非宴请）

**核心元素类型**
- 座位（Seat）
- 桌（Table）
- 入口门（Entrance）
- 主席位/主位参考点（HostSeatAnchor）
- 投影屏/主屏（Screen）
- 过道/禁入区（Aisle/NoGo，选做）

**交互**
- 拖拽移动（必做）
- 吸附开关（必做）：网格吸附 + 元素边界/中心线吸附（第一版可简化）
- 缩放/平移（必做）
- 撤销/重做（建议必做，至少10步）
- 元素属性面板（必做）：类型、朝向、座位数、是否锁定位置、是否参与规则计算

**验收标准**
- 任意元素可拖拽到新位置，释放后不抖动、不穿透
- 开启吸附时，移动到阈值范围内自动对齐并有视觉提示
- 画布缩放后坐标系一致，导出图片与屏幕所见一致（WYSIWYG）

---

### 4.3 场景选择（Strategy Picker）
**结构**
- 会议类型：汇报/评审/签约/座谈/培训（MVP可先做3个）
- 接待情景：客户到访/上级视察/政府接待/内部会议

**产出**
- strategyId（策略枚举）
- strategyParams（可选参数：是否需要主持人、是否需要汇报人靠近屏幕等）

**验收标准**
- 同一项目可随时切换策略并重新推荐（保留手动锁定位）

---

### 4.4 人员排序（People）
**人员字段（MVP）**
- name（必填）
- side（必填）：主宾方/主人方
- rank（必填）：职级（数字越大越高，或等级枚举映射为数字）
- roles（选填，多选）：主宾、最高领导、主持人、汇报人、随行/陪同
- note（选填）：忌讳/行动不便/需要靠边等

**能力**
- 手动新增/编辑/删除
- 批量导入CSV（见第6节）

**验收标准**
- 至少支持 30 人录入并能生成推荐（具体上限可由会员权益控制）
- 输入校验清晰（空姓名、非法rank、未知side等）

---

### 4.5 快速排座（Rules Engine + Placement）
**输入**
- 画布：定位元素与座位集合（含坐标、朝向、所属区域）
- 人员：side/rank/roles
- 策略：strategyId + params
- 用户锁定：lockedAssignments

**输出**
- assignments：seatId → personId
- explain：每个关键人员的落位理由（简版即可）
- warnings：风险提示（如“主宾背对投影”“主位参考点缺失”等）

**MVP规则设计原则**
- 规则分两类：
  1) **硬约束（Constraint）**：不满足就判无解或强制重排（如主宾必须在主位侧/主区）
  2) **软偏好（Scoring）**：以打分排序实现（如越靠近主位得分越高）
- **可解释**：每个落位至少给出 1–3 条命中理由
- **可覆盖**：用户手动拖拽后的 seat 锁定，再次推荐不改变锁定项

**验收标准**
- 在模板完整（入口/主席位/屏幕齐全）的情况下，点击快速排座可一次生成结果
- 锁定任意座位后再次推荐，锁定座位不变化
- 缺失关键定位元素时给出明确提示与降级策略（例如仅按rank排序入座）

---

### 4.6 导出模块（Export）
**导出PNG**
- 内容：座位号 + 姓名（可开关）+ 阵营颜色（可选）
- 清晰度：标清/高清

**付费联动**
- 免费：导出标清（可加水印）+ 排队10–20秒
- 订阅/按次：高清无水印立即导出（或较短等待）

**验收标准**
- 导出图与画布一致（位置、文本不偏移）
- 导出失败可重试，并给出失败原因（权限/存储空间等）

---

### 4.7 登录、会员与支付（Auth + Membership + Payment）
**账号**
- 微信登录（openid/unionid视产品需要）
- 用户基础信息：uid、昵称、头像（可选）

**会员体系**
- 状态：free / subscribed（到期时间） / per_export_balance（可选，按次购买余额）
- 权益校验：在导出前、导出中都应校验（防绕过）

**支付**
- 订阅：月/年
- 按次：购买“高清导出次数包”或“单次导出”

**验收标准**
- 支付成功/失败/取消均有明确反馈
- 订单回调幂等处理（同一订单多次回调不重复发放权益）

## 6. CSV导入（V1必须）

### 6.1 模板文件：`people_import_template.csv`
**字段（推荐顺序）**
1. name（必填）
2. side（必填）：`guest`（主宾方）/ `host`（主人方）
3. rank（必填）：整数，数值越大职级越高（例如 10 > 9）
4. roles（选填）：逗号分隔，支持 `vip,leader,hoster,presenter,assistant`
5. note（选填）

**示例**
```csv
name,side,rank,roles,note
张三,guest,10,vip,主宾
李四,host,9,leader,我方最高领导
王五,host,7,hoster,主持人
赵六,guest,8,,随行
```

### 6.2 校验与错误处理
- 逐行校验：name为空/side非法/rank非整数 → 标记该行失败并给出原因
- 允许“部分成功导入”：展示成功/失败数，并支持导出失败行报告（V1可选）

## 7. 技术架构原则（最小耦合 / 模块化）

### 7.1 分层与依赖方向（推荐：六边形/端口适配）
- **domain（纯业务）**：规则引擎、数据模型、排座算法（不依赖小程序API）
- **application（用例层）**：新建项目、导入、快速排座、导出等用例编排
- **adapters（适配层）**：
  - wechat-adapter：文件选择、图片保存、登录、支付
  - storage-adapter：本地缓存/云端存储
  - canvas-adapter：渲染与交互（可替换实现）
- **ui（界面层）**：页面、组件、状态管理

依赖规则：`ui -> application -> domain`，adapters 通过接口注入，domain 不反向依赖。

### 7.2 可测试性策略
- domain 层全部可在 Node 环境跑单测（无需小程序运行时）
- 规则引擎函数设计为**纯函数**：输入确定 → 输出确定（便于回归）

## 12. 默认排座策略（v1：上级视察/客户到访/政府接待）

### 12.1 主位判定（系统自动 + 用户可改）
**目标**：无须用户理解礼仪细节也能快速落到“可用主位”，但保留强制修正的入口。

主位候选优先级（按模板与定位元素存在情况）：
1. 用户在画布中显式标记的「主席位/主位点」（最高优先级）
2. 主席台中心位（若模板包含主席台）
3. 面向投影屏（Screen）的前排中心位（若存在屏幕且模板为面向式）
4. 模板预设主位（fallback）

**交互要求**
- 画布提供「主位/主位区」编辑模式：可点选某个座位设为主位；或框选/点选主位区（用于多候选）
- 用户修改主位后：后续“快速排座/重新推荐”都以用户设定为准

### 12.2 主宾/上级指定（强约束）
**MVP规则**：主宾/上级必须由用户勾选（例如 roles 中的 `vip`/`leader`），系统不做“自动猜测主宾”。

**验收标准**
- 若策略为“上级视察/客户到访/政府接待”且未勾选主宾：按钮置灰或弹窗提示“请先指定主宾/上级”
- 若勾选多个主宾：允许但需二次确认（会影响主位区分配）

### 12.3 规则引擎实现形态（可解释、可覆盖）
规则分两类：
- **硬约束（Constraint）**：不满足则强制调整或报错提示（MVP优先强制调整）
- **软偏好（Scoring）**：打分排序，用于在可行解中选择“更符合策略”的落位

并提供：
- explain：为关键人员生成 1–3 条命中理由（用于“为什么这样排”）
- warnings：定位元素缺失/冲突时的风险提示
- lockedAssignments：用户锁定座位后重排不改变锁定位

### 12.4 三种策略的默认规则（第一版清单）
#### A) 上级视察（最保守正确）
硬约束：
- 上级（vip/leader）必须在主位/主位区
- 主人方最高职级必须在上级相邻关键位（左右一）
软偏好：
- 汇报人/主持人靠近投影屏且靠近上级的可控场位置
- 非关键人员尽量外侧或后排

#### B) 客户到访（客户体验优先）
硬约束：
- 客户主宾（vip）必须在主位/主位区
软偏好：
- 我方接待负责人/主持人偏向入口侧（便于引导与控场）
- 客户核心人员靠近主位；我方同级或略低级人员坐其两侧便于沟通

#### C) 政府接待（强调对称与稳妥）
硬约束：
- 政府主宾（vip）必须在主位/主位区
软偏好：
- 尽量保持对称观感（同级对称或相邻）
- 避免主宾背对入口/过道干扰（若入口明确）
